# 오늘의 경마운세 (Today's Racing Fortune)

## 1. 개요

### 1.1 목적

- **홈 메인 화면**에 "오늘의 경마운세" 섹션을 두어, 사용자가 앱을 열었을 때 가벼운 재미·동기 부여 요소를 제공합니다.
- **사행성 없음**: 베팅 권유·배당 예측이 아닌, **오락·참고용** 콘텐츠로만 구성합니다.
- 서비스 정체성(정보·분석 제공)과 조화되도록, "오늘 한 줄 운세" 수준의 짧은 메시지와 행운 요소(숫자·색 등)를 노출합니다.

### 1.2 서비스 정체성 준수

| 원칙                | 적용                                                        |
| ------------------- | ----------------------------------------------------------- |
| 베팅/배팅 기능 없음 | 운세는 "참고용", "재미용" 문구로 명시. 배당·베팅 권유 없음. |
| 40–60대 가독성      | 폰트 16px, 짝수 px, 명확한 대비.                            |
| 정보 서비스         | AI 분석·예측과 별도로, 경마 관련 한 줄 메시지로만 제공.     |

## 2. 기능 범위

### 2.1 운세 구성 — 4가지 항목

오늘의 경마운세는 **4가지 케이스(항목)**로 구성합니다. 각 항목마다 별도 문구 풀을 두고, 랜덤으로 하나씩 뽑아 조합합니다.

| #   | 항목          | 라벨(UI)    | 설명                                    | 예시 문구                                         |
| --- | ------------- | ----------- | --------------------------------------- | ------------------------------------------------- |
| 1   | **종합**      | 종합        | 오늘 하루 전체 운세 한 줄 요약          | "오늘은 차분히 분석한 뒤 선택하는 것이 좋습니다." |
| 2   | **경주**      | 경주 운세   | 오늘 경주 관람·분석·선택과 관련된 한 줄 | "경주 전 데이터를 꼼꼼히 보는 날이면 좋겠습니다." |
| 3   | **행운 요소** | 행운의 요소 | 행운의 번호·색·키워드 (참고용)          | 번호 3, 7 / 색 빨강 / 키워드 "집중"               |
| 4   | **조언**      | 오늘의 조언 | 마음가짐·태도 관련 한 줄                | "여유를 갖고 판단하면 좋은 하루가 됩니다."        |

- **종합·경주·조언**: 각각 별도 **문구 풀**에서 랜덤 1개 선택. 풀은 최소 20~30개 이상 권장.
- **행운 요소**: 기존처럼 행운 번호(1~2개), 행운의 색, 키워드 각각 풀에서 랜덤 선택.
- **행운의 번호·색**: 실제 베팅 권유가 아님을 명시. "경주 관람 시 참고용" 등 짧은 안내 문구 포함.

### 2.2 노출 위치

- **홈 (`/`)**: `DateHeader` 직후 또는 직전, 한 줄로 읽기 쉬운 카드/배너 형태.
- **우선순위**: 메인 폴드 위쪽에 작은 블록으로 배치해, 스크롤 없이도 "오늘의 운세"가 보이도록 합니다.

### 2.3 노출 조건

- **로그인 사용자**: "오늘의 경마운세" 섹션을 노출. DB에 저장된 해당 유저의 오늘 운세를 조회하여 표시. 없으면 생성 후 저장하고 표시.
- **비로그인 사용자**: 해당 섹션을 **보이지 않음**. 운세 없어도 됨. API 호출하지 않음.
- **경주 유무와 무관**: 로그인 시 오늘 경주가 없어도 운세는 표시 가능.

## 3. 데이터 계획

### 3.1 저장 규칙 (필수)

- **유저별·날짜별 1건**: 각 유저는 **하루에 하나의 운세**만 가짐. DB에는 `(userId, date)` 기준으로 1행.
- **날짜별 덮어쓰기**: 같은 날짜에 대해 다시 저장할 때는 **기존 행을 갱신(overwrite)**. 새로 생성하면 그날짜 레코드를 upsert.
- **랜덤 생성**: 유저가 오늘 운세를 처음 요청할 때, 템플릿 풀에서 **랜덤**으로 문구·번호·색·키워드를 뽑아 저장 후 반환. 같은 유저가 같은 날 다시 보면 **DB에 저장된 값**을 그대로 보여줌.

### 3.2 DB 스키마 (안)

- **테이블**: `user_daily_fortune` (또는 `UserDailyFortune`)
- **유니크**: `(userId, date)` — 한 유저당 날짜당 1행. `date`는 `YYYYMMDD` 문자열 또는 `Date` 타입.

| 컬럼           | 타입              | 설명                              |
| -------------- | ----------------- | --------------------------------- |
| id             | PK                | 자동 증가                         |
| userId         | FK → User         | 사용자 ID                         |
| date           | string (YYYYMMDD) | 해당 날짜                         |
| messageOverall | string            | **종합** 운세 한 줄               |
| messageRace    | string            | **경주** 운세 한 줄               |
| messageAdvice  | string            | **오늘의 조언** 한 줄             |
| luckyNumbers   | Json (number[])   | **행운 요소** — 번호 배열 [3, 7]  |
| luckyColor     | string            | **행운 요소** — "빨강", "파랑" 등 |
| luckyColorHex  | string?           | KRA 게이트 색 매핑용 (선택)       |
| keyword        | string?           | **행운 요소** — 오늘의 키워드     |
| createdAt      | DateTime          | 최초 생성                         |
| updatedAt      | DateTime          | 마지막 갱신(덮어쓰기 시)          |

- **저장 시**: `userId` + `date`로 upsert. 있으면 update, 없으면 insert. → **날짜별로 덮어씌어지기**.
- **4가지 항목**: 종합·경주·조언은 각각 문구 풀에서 랜덤 1개, 행운 요소는 번호/색/키워드 풀에서 각각 랜덤 선택 후 한 세트로 저장.

### 3.3 생성 방식 (랜덤 + 풀, 4가지 케이스)

- **4가지 항목별 풀**:
  - **종합**: 문구 풀 A (20~50개) → `messageOverall`
  - **경주**: 문구 풀 B (20~50개) → `messageRace`
  - **조언**: 문구 풀 C (20~50개) → `messageAdvice`
  - **행운 요소**: 번호 후보(1~12 등에서 2개), 색 후보, 키워드 후보 → 각각 랜덤 선택
- **랜덤**: 각 풀에서 독립적으로 무작위 선택 (서버 난수).
- **저장 시점**: 유저가 "오늘 운세"를 조회했을 때, 해당 날짜 레코드가 없으면 **4가지 모두** 랜덤 생성 후 DB에 한 행으로 저장(upsert). 이후 같은 날에는 DB 값만 반환.

### 3.4 API·데이터 구조

```ts
interface TodaysRacingFortune {
  date: string; // YYYYMMDD
  messageOverall: string; // 종합
  messageRace: string; // 경주 운세
  messageAdvice: string; // 오늘의 조언
  luckyNumbers: number[]; // 행운의 번호
  luckyColor: string;
  luckyColorHex?: string;
  keyword?: string;
}
```

- **GET /api/fortune/today** (또는 `/api/fortune?date=YYYYMMDD`)
  - **인증**: 로그인 사용자만. 비로그인 시 401 — 클라이언트는 이 API를 호출하지 않고, 운세 섹션 자체를 렌더하지 않음.
  - **동작**: (1) DB에서 해당 유저 + 오늘 날짜 조회. (2) 있으면 그대로 반환. (3) 없으면 랜덤 생성 → DB upsert (날짜별 덮어쓰기) → 반환.
- **응답**: `TodaysRacingFortune`. 클라이언트는 로그인 시에만 호출하고, 이 값을 홈에서 그대로 표시.

## 4. UI/UX 요구사항

### 4.1 컴포넌트

- **이름 제안**: `TodaysFortuneCard` 또는 `RacingFortuneBanner`.
- **스타일**: 기존 홈 카드·테마(다크 골드, stone 계열)와 통일.
- **4가지 항목 노출**:
  - **종합**: 카드 상단 또는 첫 줄에 강조 (예: "오늘의 종합 운세")
  - **경주 운세**: 라벨 + `messageRace` 한 줄
  - **오늘의 조언**: 라벨 + `messageAdvice` 한 줄
  - **행운의 요소**: 행운 번호·색·키워드를 한 블록으로 (아이콘/뱃지 등)
- **접근성**: 제목은 `h2` 또는 `section` 내 `aria-label`로 "오늘의 경마운세" 명시.

### 4.2 필수 문구 (법적·정책)

- 카드 하단 또는 툴팁에 **반드시** 포함:
  - "참고용·오락 목적이며, 베팅 권유가 아닙니다."
- 필요 시 [LEGAL_NOTICE.md](../legal/LEGAL_NOTICE.md)와 링크로 연결.

### 4.3 반응형

- **모바일**: 한 줄 + 행운 번호/색을 한 블록으로, 터치 영역 충분히.
- **데스크톱**: DateHeader 옆이나 아래 한 줄 배치로 홈 레이아웃과 조화.
